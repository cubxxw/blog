# 把学习变成一个 AI 项目:目的驱动的目录结构 + 多平台发布方法论

> 深度调研报告(第二部分)· 2026 年 7 月
> 承接《AI 时代深度学习与行业调研方法论》。这一份聚焦三件事:**(1)** 各家 AI 的 Project/知识库工具怎么选;**(2)** 如何围绕「一个明确目的」把学习组织成一个可持续维护的目录/项目;**(3)** 学完之后如何按平台改造发布——而讲解与发布本身,就是最高级的学习。

---

## 核心心法(先立这三条)

1. **一切服务于目的。** 学习不是为了囤积信息,是为了「创造」。学营销,目的是「让我的产品卖得更好」;所以整个项目的目录、资料、方法论,都应当围绕这个目的组织,而不是按学科教材的章节堆。目的越具体,目录越清晰,AI 越好用。

2. **把学习做成「项目」,不是「聊天」。** 网页版聊天里的资料和产出是留不下来的。真正能沉淀、能复利的,是一个**有目录结构、能被 AI 直接读写、能版本化维护**的知识仓库——你和 AI 一起长期经营它,就像雇了一支专家团队。

3. **发布是「第二次深度加工」。** 把一份深度内容改造成公众号/小红书/视频,不是搬运,而是逼你把知识点吃透到「能用一句钩子讲清楚」。费曼说过,讲解是检验理解的终极方式。发布=对外传播 + 对内深化,一举两得。

## 第一部分:工具选型——网页版 Project vs Claude Code 文件夹工作流

你提到的 NotebookLM,本质就是「用 Project 管理一批资料」。但各家的「Project」定位差别很大,最关键的分野是:**资料是「上传到云端的只读副本」,还是「AI 能直接读写的本地文件」。**

### 网页版 Project(开箱即用,适合「读 + 生成」)

| 工具 | 一句话定位 | 强项 | 关键局限 |
|---|---|---|---|
| **NotebookLM** | 源锚定知识库 | 每条回答带引用锚定原文;一键生成播客/思维导图/测验/闪卡 | notebook 之间隔离;资料上云;是「读+生成」不是「可写回的工作区」 |
| **ChatGPT Projects** | 灵活的智能工作区 | 项目内共享文件+指令+记忆,能联网调用通用知识 | 引用溯源不如 NotebookLM;文件是喂给对话的资料,非可编辑资产 |
| **Claude Projects** | 深度推理 + 规则设定 | 项目指令优先级高、擅长起草;从论文取回原文引用较准 | 云端托管;文件只读、无版本控制、难自动化 |
| **Gemini Gems** | 可复用的「已保存专家」 | 常驻指令+参考文件,嵌进 Docs/Gmail/Workspace 流程 | 每个 Gem 仅 10 个知识文件,偏定制助手非大资料库 |
| **Notion AI** | 工作区内的 AI 助理 | 跨 Slack/Drive/GitHub 按权限统一检索 | 知识以 Notion 数据库托管,非本地纯文本、非 git |

### Claude Code 文件夹工作流(可维护、可版本化,适合「长期经营」)

这正是你直觉到的方向——**Claude Code 绑定一个本地文件夹作为工作区,能直接读、建、改目录里的真实文件**,项目规则写在 `CLAUDE.md`(随项目存放、开会话自动读取、可 git diff)。它和网页版 Project 的本质差异:

| 维度 | 网页版 Project | Claude Code 文件夹工作流 |
|---|---|---|
| 资料形态 | 上传到云端的**只读副本** | 本机**真实文件**,可被 AI 直接读写改 |
| 产出归属 | 留在对话里,难沉淀 | 就是磁盘上的 Markdown,**你拥有** |
| 记忆/规则 | 隐式项目记忆 | 明文 `CLAUDE.md`,可审阅可 diff |
| 版本控制 | 无 | 全流程 git 可回滚、可协作 |
| 自动化 | 基本无 | hooks / skills / subagents 自动整理提交 |
| 结构 | 扁平资料袋 | 可维护整个目录结构(PARA 等) |

> **一句话**:网页版是「把资料喂给 AI 问答」;Claude Code 是「AI 进驻你的目录,和你一起长期读写、版本化、自动化维护一个真实知识仓库」。

### 务实的组合用法(不用二选一)

- **NotebookLM** 做「一批资料 → 引用问答 + 播客/测验」的快速消化;
- **ChatGPT / Claude Projects** 做灵活的对话式加工与起草;
- **Claude Code(+ Obsidian / Git)** 做长期沉淀、可版本化、可自动化的领域知识库——这是你的「主仓库」,其余是「临时工作台」。

一个成熟的开源思路是 Claude Code + Obsidian:笔记是你磁盘上的纯 Markdown(local-first、你拥有数据),AI 帮你把课程/书/论文/视频处理成结构化互链笔记,还能用 hook 做 Git 自动提交。

## 第二部分:目的驱动的项目化学习(以「营销」为例)

你描述的流程——先定目的、再组织目录、然后持续维护——正是 2026 年 PKM(个人知识管理)的主流范式:**从「数字图书管理员」(手工打标签、嵌套文件夹)转向「创意总监」(让 AI 提议目录结构,并为每个项目生成 context 文件)。** 底层仍是经典的 PARA(Projects/Areas/Resources/Archives)+ CODE(Capture/Organize/Distill/Express)。

### Step 0:把目的写成一句话(最重要的一步)

不要写「学营销」。写:

> **「为 XX 产品建立一套可复用的营销方法论,3 个月内验证 2 个能带来真实增长的获客渠道。」**

这句话会成为 `CLAUDE.md` 的第一行。之后 AI 的每一次整理、每一条建议,都围绕它对齐。

### Step 1:一个围绕目的的目录结构

```
marketing-project/
├── CLAUDE.md                      # 项目宪法:目的、你的产品、语气、维护规则
├── 00-目的与路线图/
│   └── roadmap.md                 # 目标拆解、里程碑、当前卡点
├── 01-学习-营销知识/               # 「学」:领域知识的输入与消化
│   ├── 定位与品牌.md
│   ├── 获客渠道/
│   └── 转化与留存.md
├── 02-最佳实践库/                  # 「参考」:值得抄的案例、拆解、模板
│   ├── 竞品营销拆解/
│   └── 爆款案例/
├── 03-我的产品信息/                # 「我方事实」:AI 生成方案的 grounding 源
│   ├── 产品定位与卖点.md
│   ├── 用户画像与访谈.md
│   └── 历史数据与渠道表现.md
├── 04-方法论产出/                  # 「创造」:结合前三者提炼出的自有方法论
│   └── 我的营销方法论-v1.md
└── 05-发布/                        # 「传播」:分平台改造后的成品
    ├── 公众号/
    ├── 小红书/
    └── 视频脚本/
```

这五个目录对应 CODE 的闭环:**01 学(Capture/Organize)→ 02 参考 + 03 我方事实(Distill 的原料)→ 04 方法论(Distill)→ 05 发布(Express)**。目的(00)统领全部。

### Step 2:让 AI 当「专家团队」而不是搜索框

在 `CLAUDE.md` 里写清规则,AI 每次开工都遵守。例如:

- 「你是我的营销顾问团队。所有方案必须 grounding 在 `03-我的产品信息/`,不得脱离我的真实产品空谈。」
- 「引用最佳实践时,注明来自 `02-最佳实践库/` 的哪个案例,并说明为什么适用于我的产品。」
- 「更新 `04-方法论产出/` 时保留旧版本,新增内容标注日期和依据。」

这样,你不是在「问 ChatGPT」,而是在**指挥一支有记忆、有资料、有纪律的顾问团队**,持续维护一个越用越值钱的仓库。

### Step 3:持续维护(项目的复利来自这里)

- **每次学到新东西**:先自己写下理解(保住「生成效应」),再让 AI 挑错补充,存入 01。
- **定期蒸馏**:让 AI 对比 01/02/03,更新 04 的方法论,标出「新证据推翻了哪条旧结论」。
- **版本化**:用 git 提交,每次维护留痕,方法论的演化本身就是你的资产。

> ⚠️ 呼应第一部分的警示:让 AI 做「找料、出题、挑错、整理」,把「判断哪条适合我的产品、下一步做什么」留给自己。否则会陷入「认知卸载」——方案看着漂亮,但你没长出真本事。

## 第三部分:发布——一次深度创作,多平台原生分发

学完了、方法论有了,接下来是「第二次作用力」:把深度内容改造成不同平台的形态。核心原则是 **COPE(Create Once, Publish Everywhere)——但是「原生适配」,不是「复制粘贴」**。2026 年各大平台算法都能识别机械搬运并限流,所以每个平台都要重写首段、换标题、调结构。

### 先做选题:内容缺口分析(找蓝海)

发之前先问:**哪些话题「搜索需求高 × 优质供给少」?** 那才是值得投入的选题。

- 看平台**下拉联想词**里「有需求但没人做好」的词——那是占位机会。
- 工具:通用用 Ahrefs Content Gap / Semrush;中文/小红书用 5118(长尾词挖掘、按笔记数看竞争度)、千瓜、新红。
- 判断一个词要综合看**搜索量 + 相关性 + 商业价值 + 竞争度**,别只看热度。

### 母资产 → 三平台的结构对比

假设母资产是你 `04-方法论产出/` 里的《我的营销方法论》。

**① 公众号(承载「深度」,做首发母体)**

- 篇幅 800–2000 字为完读黄金区,信息密度优先;深度可到 3000 字。
- 结构:悬念/数字型标题(≤30 字,**别夸大**,平台对标题党降权)→ 提问式开头 3 秒抛痛点 → SCQA 主体 + 小标题 + 加粗,每 300 字一个钩子 → 结尾号召「在看/转发」(激活社交裂变权重)。

**② 小红书(承载「实用清单 + 搜索」)**

- 把长文拆成**可执行清单**(「5 个避坑」「3 步搞定」)——实用价值高的内容最易被收藏转发。
- 结构:大字号封面(痛点/结果前置)→ 标题 ≤20 字,前 18 字含 2 个核心词 → 正文 600 字+ → 标签用 **3 长尾 + 2 热门 + 1 个人 IP**。
- 搜索是主流量入口,标题和正文前部埋核心词(但别堆砌)。

**③ 口播短视频(承载「钩子 + 引流」)**

- 时长 30–60 秒起步;抓长文里**最尖锐的 1 个点**做流量入口,引回长文。
- 脚本 = **钩子—价值—引导**:0–3 秒制造「认知缺口」(反常识/痛点)→ 3–33 秒 1 个观点 + 1 个场景案例(讲场景不讲道理)→ 33–50 秒对比/破误区 → 50–60 秒 CTA(关注看完整版)。

> 一句话:**公众号承载深度,小红书承载实用清单+搜索,视频承载钩子+引流;三者共享方法论内核,但标题、开头、结构、篇幅各自原生化。**

### 让内容「被主动分享」:STEPPS 六要素

什么内容用户愿意帮你传播、点赞、二创?Jonah Berger 的 STEPPS 框架(有传播学研究支撑,比多数「爆款公式」可靠)给了答案:

- **Social Currency 社交货币**:让分享者显得内行/有品;
- **Triggers 触发物**:绑定高频场景,让人常被提醒想起;
- **Emotion 情绪**:高唤醒情绪(敬畏、兴奋、愤怒、幽默)驱动分享;
- **Public 公开性**:可见的行为更易被模仿;
- **Practical Value 实用价值**:有用的清单/教程因「利他+互惠」被主动转发;
- **Stories 故事**:把观点包进让人无法绕开你核心信息的故事里。

围绕「实用价值 + 情绪 + 社交货币」搭内容,比追逐单个「爆款套路」可持续得多。

### 发布也回流到学习(闭环)

把每个平台的成品存回 `05-发布/`,并让 AI 分析数据反馈(哪条钩子留存高、哪个选题搜索多但你没覆盖),回头补进 04 方法论和下一轮选题。**发布 → 数据 → 迭代方法论**,项目就活起来了。

## 一张图收尾:整个闭环

```
       ┌─────────────── 目的(一句话)───────────────┐
       │                                              │
   [学] 01 营销知识 ──┐                                │
   [参考] 02 最佳实践 ─┼──► [蒸馏] 04 我的方法论 ──► [发布] 05 分平台改造
   [我方] 03 产品信息 ─┘         ▲                         │
       │                        │                         │
       │                   数据反馈 ◄───────────────────────┘
       │             (哪条钩子有效 / 缺口选题)
       └──────────── Claude Code + git 持续维护 ────────────┘
```

**工具落位**:主仓库用 Claude Code 管这个目录 + git 版本化;NotebookLM 快速消化一批资料出播客/测验;ChatGPT/Claude Projects 做灵活起草;发布前用 5118/Ahrefs 做缺口分析。
