# ✍️ Writing Pipeline — 单一母版，多平台分发

> 一句话：**博客不是源头，母版才是。** 内容只写一次（母版），其余平台全部由母版派生。

这个目录放在 Hugo 站点根目录，但 **不会被发布**（Hugo 只渲染 `content/` `static/` `assets/` 等，不碰这里）。它是你所有内容的"上游"，博客只是它的众多下游之一。

---

## 五层流水线

```
0-inbox/      收集箱      →  零摩擦捕获灵感/碎片，一条就是一条，绝不整理
1-notes/      原子笔记    →  学习时把知识点系统化，每条自包含、可互链
2-master/     母版(SSOT)  →  一次写完的完整版，纯 Markdown，平台无关 ★真相源★
3-adapters/   适配器      →  AI 指令模板，把母版派生成各平台版本
4-published/  发布追踪    →  哪篇发到哪、表现如何，反哺选题
```

数据只朝一个方向流：**碎片 → 原子笔记 → 母版 → 各平台**。永远不要反过来从博客往外抠内容。

---

## 每一层怎么用

### 0-inbox — 收集箱
随手记，怎么快怎么来。一行字、一个念头、一个链接都行。**在这里绝不整理、绝不纠结格式**。定期（比如每周）回顾，把值得展开的挑进 `1-notes/`，其余归档或删掉。

### 1-notes — 原子笔记
学 AI / 看资料时，把每个知识点写成一条**自包含**的笔记（脱离上下文也读得懂）。笔记之间用 `[[双链]]` 互相引用。当同一主题的原子笔记攒够，就能聚合升级成一篇母版——这就是你说的"整理成一系列学习笔记"。

### 2-master — 母版（唯一真相源）
用 `_TEMPLATE.md` 起一篇，写**完整、深入、平台中立**的版本。不带任何 Hugo 专属语法、不迁就任何平台的字数或调性。frontmatter 里填好给适配器读的元数据（钩子、人群、痛点、硬核结论、关键词、要分发到哪些平台）。这是你说的"第一版学习笔记"，它是别的一切的母亲。

### 3-adapters — 适配器（AI 自动化核心）
每个平台一个指令文件，封装了那个平台的**算法机制 + 人群心理**。用法统一：

> "读 `2-master/<某篇>.md`，按 `3-adapters/xiaohongshu.md` 的规则生成小红书版本。"

我（Claude）会照着适配器把母版转成对应平台的成品。四套已备好：`blog-hugo` `xiaohongshu` `overseas-tech` `video-script`。

### 4-published — 发布追踪
`tracker.md` 记录每篇母版分发到了哪些平台、链接、日期、数据表现。回看哪类选题在哪个平台跑得好，指导下一步写什么。

---

## 一次完整流程示例

1. 平时刷到一个想法 → 丢进 `0-inbox/`
2. 学某个 AI 主题，写了 5 条原子笔记进 `1-notes/`
3. 攒够了，聚合成 `2-master/rag-from-scratch.md`（完整版 + 元数据）
4. 对我说："把这篇母版分发到博客、小红书、Dev.to、B站脚本"
5. 我按四套适配器生成四个版本，博客版直接落到 `content/zh|en/...`
6. 你发布后，在 `4-published/tracker.md` 记一行

---

## 唯一要守的纪律

**改内容只改母版。** 发现博客版有错？回去改母版，再重新分发。否则各平台版本会各自漂移，母版就名存实亡了。

---

## 以后可以怎么长大

- 母版量大了，可以把 `writing-pipeline/` 抽成独立仓库（内容与站点解耦）。
- `1-notes/` 可以接 Obsidian vault，享受双链和图谱。
- 分发可以做成定时任务：写完母版自动提醒你分发。
- 适配器可以继续加：公众号、知乎、掘金、Newsletter……照现有格式复制一份即可。
